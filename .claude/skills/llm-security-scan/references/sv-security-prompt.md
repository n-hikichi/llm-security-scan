# セキュリティレビュープロンプト

あなたはセキュリティ専門家として、Node.js/Express サーバーアプリケーションの包括的なセキュリティレビューを実施してください。

## レビュー対象
- プロジェクト: heresme-sv (Here's Me Server Application)
- 技術スタック: Node.js v22.14.0, Express.js 4.16.0, PostgreSQL
- 言語: JavaScript (CommonJS)

## レビュー重点領域

### 1. 認証・認可の実装
以下の観点で詳細にレビュー:
- JWT トークンの生成、検証、有効期限管理の実装
- トークンのセキュアな保存と送信（HTTPS、HttpOnly Cookie など）
- パスワード管理（ハッシュ化、ソルト、アルゴリズムの強度）
- IP アドレス認証の実装と回避可能性
- セッション管理とトークン更新のメカニズム
- 認可チェックの実装（権限管理、アクセス制御）
- ブルートフォース攻撃への対策

### 2. データベースセキュリティ
以下の観点で詳細にレビュー:
- SQL インジェクションの脆弱性（特にパラメータ化クエリの使用状況）
- データベース認証情報の管理（ハードコード、環境変数、シークレット管理）
- 最小権限の原則の適用
- データベース接続のセキュリティ（SSL/TLS の使用）
- センシティブデータの暗号化（保存時、転送時）
- データベースクエリのログ出力による情報漏洩

### 3. 入力検証とサニタイゼーション
以下の観点で詳細にレビュー:
- すべてのユーザー入力の検証（型、長さ、フォーマット、範囲）
- ホワイトリスト方式の検証の使用
- XSS（クロスサイトスクリプティング）対策
- コマンドインジェクション対策
- パストラバーサル対策
- ファイルアップロードのセキュリティ（multer の使用箇所）
- Content-Type の検証

### 4. 依存関係の脆弱性
以下の観点で詳細にレビュー:
- 既知の脆弱性を持つパッケージの特定
- パッケージバージョンの古さの評価
- 直接依存と間接依存の脆弱性
- 推奨される更新パスと互換性の問題

## 追加の重要なセキュリティ観点

### CORS とセキュリティヘッダー
- CORS 設定の適切性（`cors()` の設定）
- セキュリティヘッダーの実装状況（CSP, X-Frame-Options, HSTS など）

### エラーハンドリングと情報漏洩
- スタックトレースやエラー詳細の漏洩
- ログファイルでの機密情報の記録
- デバッグ情報の本番環境での無効化

### 暗号化とデータ保護
- **通信の暗号化（HTTPS の実装）** - **必須チェック項目**
  - `bin/www` または起動スクリプトで `https.createServer()` が使用されているか
  - `http.createServer()` のみの場合は Critical 脆弱性として報告
  - リバースプロキシ（Nginx, ALB等）でのTLS終端の有無を確認
  - 開発環境でもHTTPSを使用しているか
- センシティブデータの暗号化（保存時、転送時）
- 暗号化アルゴリズムの強度（推奨: AES-256, RSA-2048以上）
- SSL/TLS証明書の管理と有効期限

### レート制限と DoS 対策
- API レート制限の実装
- リソース消費の制限
- タイムアウト設定

### セッションと Cookie のセキュリティ
- Cookie の Secure、HttpOnly、SameSite 属性
- セッション固定攻撃への対策

## レビュー出力形式

各脆弱性について、以下の形式で報告してください：

### 脆弱性タイトル
**重要度**: Critical / High / Medium / Low

**場所**: ファイルパス:行番号

**説明**:
脆弱性の詳細な説明

**影響**:
この脆弱性による潜在的な影響やリスク

**再現手順**:
可能であれば、脆弱性を再現する手順

**修正提案**:
具体的な修正方法の説明

**修正コード例**:
```javascript
// 修正前
[問題のあるコード]

// 修正後
[推奨される修正コード]
```

**参考資料**:
関連する OWASP ガイドライン、CWE、CVE などへのリンク

## レビュー手順

1. **インフラストラクチャとサーバー起動の確認** - **最優先**
   - `bin/www` または `server.js` などの起動スクリプトを確認
   - HTTPSサーバーの実装有無をチェック
   - `http` vs `https` モジュールの使用状況
2. 各ファイルの Symbol Overview を取得
3. 認証関連ファイル（login-auth.js, api-auth.js）の詳細分析
4. データベース関連ファイル（db-util.js, models/*.js）の詳細分析
5. ルーターファイル（routes/r-*.js）での入力検証の確認
6. package.json の依存関係の脆弱性チェック
7. 全体的なアーキテクチャとセキュリティ設計の評価

## 最終レポート構成

1. **エグゼクティブサマリー**
   - 発見された脆弱性の総数（重要度別）
   - 最も重大な問題のハイライト
   - 全体的なセキュリティ評価

2. **詳細な脆弱性リスト**（重要度順）
   - Critical レベルの脆弱性
   - High レベルの脆弱性
   - Medium レベルの脆弱性
   - Low レベルの脆弱性

3. **ベストプラクティスの推奨事項**
   - セキュリティヘッダーの追加
   - 入力検証ライブラリの導入
   - セキュリティ監視とロギングの強化
   - 定期的なセキュリティレビューの実施

4. **アクションプラン**
   - 優先順位付けされた修正ロードマップ
   - 短期的な対応（1週間以内）: Critical および High レベルの脆弱性
   - 中期的な対応（1ヶ月以内）: Medium レベルの脆弱性とアーキテクチャ改善
   - 長期的な対応（3ヶ月以内）: Low レベルの脆弱性とベストプラクティスの適用

## 重要な注意事項

- すべてのコードファイル（models/, routes/, util/, config/）を対象にする
- ハードコードされた認証情報や秘密鍵を特に注意深くチェック
- 古いパッケージバージョンによる既知の脆弱性を確認
- プロジェクト固有の実装（IP 認証、JWT トークン管理など）を重点的に分析
- 修正コード例は、既存のコーディングスタイル（インデント、命名規則など）に従う

## Critical脆弱性の必須チェック項目

以下の項目は**必ず確認**し、問題があればCriticalとして報告すること:

1. **HTTPSの実装状況** (`bin/www`, `server.js`)
   - `require('https')` と `https.createServer()` の使用確認
   - HTTPのみの場合は即座にCritical脆弱性として報告

2. **ハードコードされた認証情報** (全ファイル)
   - JWT秘密鍵、APIキー、パスワード、トークン
   - 特に `config/`, `models/db-util.js` を重点確認

3. **JWT署名検証の欠如** (`models/api-auth.js`)
   - `jwt.decode()` vs `jwt.verify()` の使用状況
   - 署名検証なしのトークン処理

4. **SQLインジェクション** (`models/db-util.js`, `routes/`)
   - 文字列連結によるクエリ構築
   - パラメータ化クエリの未使用

5. **依存関係の脆弱性** (`package.json`)
   - Critical/High脆弱性を含むパッケージ
   - 依存関係の脆弱性チェック結果を必ず含める
