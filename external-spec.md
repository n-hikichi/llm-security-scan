# LLM Security Scan — 外部仕様書

> Version: 1.0.0 | Date: 2026-03-14

## 概要

Claude AI の言語理解能力を活用し、JavaScript/Node.js プロジェクトのソースコードを直接分析して脆弱性を検出するセキュリティスキャン SKILL。プロジェクトタイプ（クライアント／サーバー）を自動判定し、適切なセキュリティチェックリストに基づいてレビューとレポートを生成する。

## 対応言語・拡張子

| 拡張子 | 言語 | 対応状況 |
|--------|------|----------|
| `.js`  | JavaScript (ES6, CommonJS) | ✅ 対応 |
| `.jsx` | JSX (React) | ✅ 対応 |
| `.ts`  | TypeScript | ✅ 対応 |
| `.tsx` | TSX (React + TypeScript) | ✅ 対応 |
| `.vue` | Vue SFC | ✅ 対応 |
| `.mjs`, `.cjs` | ESM/CJS | ⚠️ 未検証 |

**注**: AI がソースコードを直接読み取って分析するため、パーサーやプラグインの制約を受けない。原理上、任意の JavaScript/TypeScript 記法に対応可能。

## 対応フレームワーク

| 種別 | フレームワーク | 自動検出 |
|------|--------------|----------|
| Client-side | Vue.js, React, Angular | ✅ |
| Server-side | Express.js, NestJS, Koa, Fastify | ✅ |
| Hybrid | Next.js, Nuxt.js, Remix | ✅ |

フレームワークは `package.json` の dependencies とディレクトリ構造から自動推定される。

## 検出メカニズム

ESLint 等のルールベースツールとは異なり、Claude AI がソースコードを直接読み取り、文脈を理解した上で脆弱性を判定する。

| 特性 | 説明 |
|------|------|
| 分析エンジン | Claude AI（LLM による直接コード分析） |
| コード探索 | MCP Serena（Symbol Overview による効率的なナビゲーション） |
| 判定方式 | AI によるコンテキスト理解（データフロー、意図、ビジネスロジックを考慮） |
| プロンプト | クライアント用／サーバー用の専用セキュリティチェックリスト |

## 検出対象

### クライアントサイド（Vue.js, React 等）

| カテゴリ | 検出項目 |
|---------|---------|
| 認証・認可 | トークン管理（JWT）、ストレージセキュリティ、自動ログアウト |
| XSS 対策 | v-html / dangerouslySetInnerHTML、DOM 操作、入力エスケープ |
| API 通信 | HTTPS 強制、認証ヘッダー、CORS 整合性、エラー時の情報漏洩 |
| 依存関係 | 既知の脆弱性、パッケージバージョンの古さ |
| データ保護 | localStorage/sessionStorage への機密情報保存、キャッシュ漏洩 |
| ルーティング | ナビゲーションガード、Open Redirect、URL パラメータ |

### サーバーサイド（Express.js, Node.js 等）

| カテゴリ | 検出項目 |
|---------|---------|
| 認証・認可 | JWT 生成・検証、パスワードハッシュ、セッション管理、ブルートフォース対策 |
| データベース | SQL インジェクション、パラメータ化クエリ、接続セキュリティ（SSL/TLS） |
| 入力検証 | サニタイゼーション、型チェック、パストラバーサル、コマンドインジェクション |
| 暗号化 | HTTPS 実装、AES/RSA 強度、ハードコードされた秘密鍵 |
| CORS・ヘッダー | セキュリティヘッダー（Helmet）、CORS 設定 |
| エラーハンドリング | スタックトレース漏洩、ログの機密情報 |
| レート制限 | DoS 対策、API 制限、タイムアウト設定 |

## 入力

```
セキュリティスキャンを実行して
```

- 実行場所: スキャン対象の npm プロジェクトディレクトリ（`package.json` を含む）
- 前提: Claude Code CLI + MCP Serena（推奨）+ uv
- 手動指定: 自動検出が失敗した場合、「クライアントサイドの〜」「サーバーサイドの〜」と明示可能

## 出力

| ファイル | 形式 | 用途 |
|---------|------|------|
| ターミナル出力 | Markdown（日本語） | リアルタイム確認 |
| 任意のファイル | Markdown（日本語） | 「レポートをファイルに保存して」で出力 |

**注**: JSON 形式の機械可読出力は現時点では未対応。

### レポート構成

| セクション | 内容 |
|-----------|------|
| エグゼクティブサマリー | 脆弱性総数（重要度別）、最重要問題、全体評価 |
| 詳細な脆弱性リスト | 重要度順に各脆弱性の詳細（下記参照） |
| ベストプラクティス | セキュリティヘッダー、入力検証、監視強化の推奨 |
| アクションプラン | 短期（1週間）/ 中期（1ヶ月）/ 長期（3ヶ月）の修正ロードマップ |

### 脆弱性レポート項目

各脆弱性について以下を報告:

| 項目 | 説明 |
|------|------|
| 重要度 | Critical / High / Medium / Low |
| 場所 | ファイルパス:行番号 |
| 説明 | 脆弱性の詳細 |
| 影響 | 潜在的なリスク |
| 再現手順 | 可能な場合 |
| 修正提案 | 具体的な修正方法 |
| 修正コード例 | 修正前後のコード |
| 参考資料 | OWASP, CWE, CVE へのリンク |

## 処理フロー

```
Phase 1: プロジェクト検出
  detect_project_type.py → package.json + ディレクトリ構造を分析
  → client / server / both / unknown を判定（信頼度スコア付き）

Phase 2: プロンプト生成
  generate_scan_prompt.py → 判定結果に応じた セキュリティプロンプトを選択
  → cli-security-prompt.md（クライアント用）
  → sv-security-prompt.md（サーバー用）
  → 両方（ハイブリッド用）

Phase 3: コード分析
  Claude AI + MCP Serena → Symbol Overview でファイル構造を把握
  → 認証、API、DB 関連ファイルを重点分析
  → ソースコードの文脈を理解した脆弱性検出

Phase 4: レポート生成
  Claude AI → 重要度分類、修正提案、コード例を含むレポート出力
```

## 処理時間の目安

| 規模 | 目安 | 処理時間 |
|------|------|---------|
| 小規模 | 〜50 ファイル | 約 10 分 |
| 中規模 | 50〜300 ファイル | 10〜30 分 |
| 大規模 | 300 ファイル〜 | 30 分以上 |

**注**: MCP Serena の onboarding（初回のみ 5〜10 分）は別途必要。

## 制約事項

- AI がソースコードを直接分析するため、分析対象はコンテキストウィンドウに収まる範囲に限定される
- AI 判定は「AI の判断」として明記。最終判定は人間のセキュリティレビューで行うこと
- 大規模リポジトリではセッション再起動が必要な場合あり
- JSON 形式の出力（CI/CD 連携用）は未対応
- `package.json` がないプロジェクトは自動検出不可（手動指定で対応）
- ESLint のようなルールベースの網羅的検出ではなく、AI の判断に基づくため、検出項目の再現性は保証されない

## 付録 A: ESLint Security Triage との比較

同じ JavaScript/Node.js プロジェクトを対象とする姉妹ツール「ESLint Security Triage」との設計上の差異を以下に示す。

### アーキテクチャの違い

| 観点 | ESLint Security Triage | LLM Security Scan |
|------|----------------------|-------------------|
| 検出メカニズム | ESLint プラグインによるルールベース検出 + AI トリアージ | **AI がソースコードを直接分析**（プラグイン不要） |
| AI の役割 | 偽陽性の判定（トリアージ） | **脆弱性の発見そのもの** |
| 拡張方法 | ユーザーが ESLint プラグインを追加 | AI プロンプト（チェックリスト）の編集 |
| 外部ツール依存 | ESLint + セキュリティプラグイン（必須） | MCP Serena（推奨、なくても動作可能） |

### 判定カテゴリの違い

| ESLint Security Triage | LLM Security Scan |
|----------------------|-------------------|
| 🔴 危険（Dangerous）— 脆弱性が確定的 | **Critical** — 即座に修正が必要 |
| 🟡 要確認（Needs Review）— 判断材料不足 | **High** — 重大なリスク |
| 🟢 安全（Safe）— 偽陽性と判断 | **Medium** — 中程度のリスク |
| — | **Low** — 軽微なリスク・ベストプラクティス |

ESLint Security Triage は ESLint が検出した警告を 3 段階でトリアージする設計。LLM Security Scan は AI が脆弱性を直接発見し、4 段階の重要度で分類する設計。

### 出力形式の違い

| 出力 | ESLint Security Triage | LLM Security Scan |
|------|----------------------|-------------------|
| JSON（機械可読） | ✅ `eslint-triage-result.json` | ❌ 未対応 |
| Markdown（人間可読） | ✅ `eslint-security-report.md` | ✅ ターミナル出力 + ファイル保存 |
| CI/CD 連携 | ✅ JSON 経由で可能 | ❌ 未対応 |

### 使い分けの指針

| ユースケース | 推奨ツール |
|------------|-----------|
| CI/CD パイプラインへの組み込み | ESLint Security Triage（JSON 出力、自動化向き） |
| 網羅的なルールベース検出 + 偽陽性削減 | ESLint Security Triage |
| コンテキストを理解した深い分析 | LLM Security Scan（AI が意図・ロジックを理解） |
| 初回のセキュリティレビュー・全体評価 | LLM Security Scan（セットアップが簡易） |
| ルールで捕捉できないロジック脆弱性 | LLM Security Scan |
| 両方の長所を活かした多層防御 | **併用を推奨** |
